The moving scooter on your food delivery app is three separate engineering systems, not one.

Most people assume it is GPS pinging a database and the app reading it back. The reality involves a streaming architecture, a machine learning model that snaps noisy GPS to actual roads, and four separate prediction models just for the arrival time.

**The location firehose**

At peak, Swiggy has around 300,000 riders active simultaneously, each sending GPS every 3 to 5 seconds. That is roughly 75,000 GPS writes per second. The data is stale in 4 seconds and worthless tomorrow.

The naive design writes each ping as a database row and has every customer app poll for updates. This creates a database burning through enormous IO on data nobody ever queries again, while millions of polling requests pile on top.

The fix: treat location as an event stream, not a database record. Rider GPS flows into Kafka. A processor reads the stream and keeps only the latest position per order in Redis. Customers hold an open WebSocket connection. When the rider moves, the server pushes the new point down. No polling. One socket per active order instead of millions of requests every few seconds.

**Why the scooter follows the road**

Raw GPS on a phone in a dense city has scatter of 20 to 40 meters. Plot it naively and the scooter teleports through buildings and cuts across parks.

Swiggy runs map matching using GraphHopper on OpenStreetMap data with a motorcycle routing profile (Indian deliveries are almost entirely two-wheelers, so the profile includes small lanes a car routing profile would not). The algorithm underneath is a Hidden Markov Model: the true road segment is the hidden state, the noisy GPS reading is the observation, and the Viterbi algorithm finds the most likely sequence of road segments given the full GPS trail. The output is a clean trajectory that rides on actual roads and turns at corners.

Between server updates, the client interpolates. A new GPS point arrives every few seconds. The app animates the scooter at 60 fps in between by predicting position along the snapped road geometry at roughly the rider's current speed. The truth arrives sparse. The smoothness is rendered locally from that truth. They are deliberately decoupled.

**Why the timer sometimes goes up**

"Arriving in 32 min" is not one model. It is four: time to assign a rider (O2A), time for the assigned rider to reach the restaurant (first mile), wait time for the food to be cooked and packed, and time from restaurant to your door (last mile). Each runs as a separate ML model because each is driven by different signals and can be diagnosed and fixed independently when it is wrong.

The live speed data feeding the first-mile and last-mile models comes from the GPS pings of all riders currently moving through those same roads. Swiggy does not need a third-party traffic API for roads its own fleet is on. The riders are the traffic sensor.

When the predicted arrival time goes up, the model saw something real: a slowdown measured by other Swiggy riders on that road, not a guess. It updated the promise to match the new truth. That is not a bug. It is the system being honest, and honesty is what keeps support tickets and refund costs down.

**The principle underneath all three**

Stream sparse truth, render smooth fiction on the client, and never let the live path do work you can precompute. This shows up in every part of this system: Kafka buffers the firehose so the database never sees raw pings, GraphHopper precomputes road hierarchies offline so route queries run in under a millisecond at request time, and the four-leg ETA decomposition separates what can be diagnosed per leg from what would be hidden in a single blended number.

This principle applies to any real-time UI: multiplayer cursors, live dashboards, interactive maps. The expensive computation happens offline or on a slow cadence. The live path is a cheap lookup plus client-side interpolation. The user sees 60 fps. The system does far less work than that.
