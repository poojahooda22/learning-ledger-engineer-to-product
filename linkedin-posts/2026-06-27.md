Your MacBook closes at the airport. You work offline for 90 minutes. It reconnects.

In a naive collaborative editor, the app freezes. Here is the exact math on why, and what CRDTs do instead.

**The problem is O(n*m)**

Picture 5 engineers editing a shared doc during sprint planning. One loses Wi-Fi 30 minutes in.

During those 90 offline minutes:
- The offline engineer makes 1,800 block edits.
- The 4 online teammates make 5,400 combined edits.

Classic collaborative editors use Operational Transformation (OT). When you reconnect, OT must transform each of your local ops against each server op to figure out what your edits "should have been" had you stayed online.

That is 1,800 x 5,400 = 9,720,000 transform pairs.

Each transform is a function call: compare two operations, adjust positions. On a laptop CPU at 10 microseconds per call: 97 seconds of single-threaded freeze. On a 2021 iPhone: 16 minutes.

The app hangs. Users force-quit. 90 minutes of work is stuck behind a spinner.

Google Wave hit this wall. Early Confluence hit it. This is not a theoretical problem.

**Why OT ends up here**

OT was invented in 1989 for linear text. The core idea is simple: Alice inserts at position 5, Bob inserts at position 5 simultaneously, the server shifts Bob's position to 6. Correct result.

But position numbers are fragile. The moment anyone inserts a character to your left, every position to the right shifts by one. Reconciling two diverged histories means transforming every pair of operations. That is quadratic in the number of ops on reconnect.

OT also requires a central server to serialize all operations. No true offline editing, no peer-to-peer sync, and server downtime means collaboration downtime.

And OT for tree operations, like moving a Notion block or dragging a Figma layer, has no clean formulation. The academic literature has no consensus on the correct transform functions for trees. This is why collaborative Slides took years longer than collaborative Docs.

**The CRDT model: IDs instead of positions**

CRDTs (Conflict-free Replicated Data Types) change the fundamental assumption.

Instead of tracking where a character is, track what its permanent neighbors were at the moment it was inserted.

Y.js implements this via an algorithm called YATA. Each character carries the ID of its left and right neighbors at insertion time, not a position number. Those IDs never change, no matter how many other people insert or delete around them.

When two people both insert after the same character simultaneously, YATA resolves the tie by client ID (lexicographic order). Both clients apply the same rule locally. Both converge to the same result. No server needs to arbitrate.

**The reconnect, redone**

When the offline engineer reconnects after 90 minutes:

1. Their Y.js client sends a state vector to the sync server: a tiny map of the last op ID seen from each collaborator.
2. The server compares state vectors and sends back only what the client missed: 7,200 ops.
3. The client applies each op in O(1): look up the neighbor ID in a hash table, splice the character into the linked list. No transform functions. No positional recalculation.
4. All 7,200 ops applied. Done in under 2 seconds.

OT reconnect cost: O(1,800 x 5,400) = O(9.7 million).
CRDT reconnect cost: O(7,200 x 1).

The merge function is commutative. "Apply A then B" produces the same document as "Apply B then A." No server defines the canonical order. No quadratic blowup. No freeze.

**The trade-off**

CRDTs are not free. Deleted characters leave tombstones in the linked list. Future inserts that reference them as neighbors still resolve correctly. But tombstones accumulate: after 10,000 insertions and 9,500 deletions, the data structure holds all 10,000 nodes in memory, 9,500 of them invisible to the user. Garbage collection requires confirming that every replica has seen the deletion, which is manageable for closed documents and harder for open-sharing links.

For small documents with fully-online teams, OT carries lower overhead. CRDTs win decisively for long-lived documents with any offline usage at all.

**What this means for the product**

The engineering choice is what determines whether "your doc works offline" is a real feature or a spinner with a sorry message.

Y.js and Automerge are the two production CRDT libraries engineers actually deploy. Cloudflare Durable Objects (one instance per document, globally routed, durable state) are the natural sync server: they hold the canonical CRDT state, broadcast ops to connected clients, and persist the op log. If you are already on Cloudflare, multiplayer on a document is a Durable Object, not a new infrastructure layer.

That is what separates collaborative tools that actually work offline from tools that lose your work the moment Wi-Fi drops.
