Lady Gaga had 31 million followers in 2012. When she tweeted, tens of millions of phones refreshed their timelines at roughly the same moment.

Each refresh was a cache read for the key `timeline:Lady_Gaga`.

Consistent hashing routes that key to exactly one Redis node. One node handles about 100,000 to 300,000 reads per second at safe utilization. 31 million followers refreshing every 30 seconds is roughly 1 million reads per second, all hitting the same key on the same node.

The node saturates and starts dropping connections. Other nodes in the cluster sit at 8% utilization, idle, while one drowns.

Here is the part that breaks intuition: adding more cache nodes does not help.

Consistent hashing still routes `timeline:Lady_Gaga` to one node. You could have 100 nodes. 99 would be bored. One would be on fire.

Horizontal scaling helps with average load. It does not help with correlated spikes, because correlation is the problem, not capacity.

Twitter's actual fix was three mechanisms layered together:

Key replication: store the hot key as 16 copies across 16 nodes, `timeline:Lady_Gaga:0` through `:15`. Reads pick a random replica. One key spread across 16 nodes, each seeing 1/16th of the traffic. Writes update all 16 copies asynchronously. For a key written once per tweet and read 1 million times per second, 16 writes per event is trivial.

Request coalescing (singleflight in Go): when a cache key is cold and 10,000 threads all miss simultaneously, exactly one fires the database query. The other 9,999 wait for that one result. One database hit, 10,000 threads served. Go ships this as a standard library package in 12 lines of code. Nginx calls it request collapsing. Varnish calls it request coalescing. Every major infrastructure component has a version of this.

Probabilistic early expiration (XFetch): standard TTL is dangerous because every cache copy expires at the same instant, causing a correlated miss. XFetch adds randomness: a small fraction of reads refresh the cache slightly before expiry. By the time the TTL fires, the cache is already warm. No miss window forms.

The deeper lesson is about feedback loops.

Cache expires, N threads miss, N queries hit the database, database saturates, clients time out and retry, retries pile on top of the overload. This is called a metastable failure, named by Facebook engineers in a 2021 HOTOS paper. The system gets trapped in a bad state it cannot exit on its own, because the act of trying to recover (retrying requests) is also the amplification mechanism. Getting back to the good state requires the system to serve traffic well, but it cannot serve traffic because it is in the bad state.

Each fix above breaks the loop at a different point. XFetch prevents the correlated miss from forming. Singleflight collapses amplification to one query even if a miss happens. A circuit breaker stops retries from making the fire worse after the fact.

The same failure mode appears everywhere a power-law distribution collides with hash-based routing: database shards, Kafka partitions, queue workers, CDN edge nodes. One item gets almost all the traffic. One node handles that item. The node dies. The fix is always the same: detect the hot spot, replicate it, coalesce the load.

For users, the engineering decision is about which tradeoffs are invisible. Lady Gaga's tweet appearing 200ms late is invisible to 31 million people. The cache cluster collapsing is not.
