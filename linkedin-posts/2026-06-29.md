Microsoft rewrote the TypeScript compiler in Go. The VS Code codebase is 1.5 million lines. Type-check time went from 77 seconds to 7.5 seconds. Same codebase, same type rules, roughly 10x faster.

Here is the technical reason it is 10x and not 1.5x or 2x.

The original TypeScript compiler was TypeScript running on V8. Three layers of overhead stacked on top of each other.

First, JIT warmup. V8 needs to observe a function run many times before it compiles it to native code. Every cold invocation, like a CI type-check job starting fresh, pays the full JIT compilation cost before any useful type-checking begins. A Go binary starts instantly. No warmup window at all.

Second, GC pressure. V8's garbage collector is tuned for web pages: short sessions, small heaps, latency calibrated for browser frame budgets. Type-checking a million-line program builds a working set that dwarfs any web page. GC pauses become significant and unpredictable. Go's GC is designed for long-running server processes with large, stable heaps. Its pause times are short and bounded.

Third, parallelism. TypeScript 5.x type-checks files with limited parallelism, constrained by V8's single-threaded event loop. The Go rewrite uses goroutines to type-check independent files in parallel, filling all available CPU cores simultaneously.

Each of those is a different kind of tax. JIT warmup is a startup cost. GC pressure is a steady-state cost. Single-threaded execution is a throughput ceiling. Remove all three and you compound the gains.

The break is real. Custom transformer plugins that called into the TypeScript Compiler API will fail. That API was a TypeScript-land interface that no longer exists at runtime in the Go binary. tsserver still speaks the Language Server Protocol, but over a different binary. Editor extension authors have until general availability, roughly mid-July, to test against the RC.

The deeper lesson is not about Go versus TypeScript. It is about the performance ceiling of any tool written in its own runtime.

The compiler has to be correct about the language. The runtime overhead has nothing to do with correctness. For a toolchain invoked millions of times a day across CI pipelines, that runtime tax compounds into wall-clock time no amount of algorithmic improvement can fully offset. The only way around it is to go below the language layer.

The Playwright test suite (26,000 lines) went from 25 seconds to 3 seconds. For any team with a monorepo, a 10x cut in type-check time directly changes whether full type-checking per PR is affordable or something you skip to protect CI speed.
