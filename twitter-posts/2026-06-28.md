The single cache key that nearly took down Twitter.

Lady Gaga: 31M followers in 2012. One tweet, roughly 1M cache reads per second, all routing to one Redis node.

That node's safe capacity: 300K reads/sec. 99 other nodes sat at 8% utilization while one collapsed.

Counter-intuitive: adding more cache nodes does NOT fix this.

Consistent hashing still routes `timeline:Lady_Gaga` to one node. You could have 100 nodes. 99 are idle. One burns. Horizontal scaling helps average load. Correlated spikes are a different problem.

Twitter's actual fix, two core mechanisms:

1. Key replication: store `timeline:Lady_Gaga` as 16 copies across 16 nodes. Reads pick a random replica. Each node sees 1/16th of the traffic. Writes update all 16, cheap because writes are rare.

2. Request coalescing (singleflight): when 10,000 threads all miss the cache at the same instant, exactly 1 fires the database query. The other 9,999 wait for that result. One DB hit, 10,000 threads served. 12 lines in Go. Every major infrastructure tool has a version of this.

The failure mode has a name: metastable failure.

Cache expires, N queries hit DB, DB saturates, clients retry, retries add more load. The system cannot recover because recovery requires serving traffic and it cannot serve traffic. External intervention needed.

Both mechanisms above break that feedback loop before it forms. That is the actual fix, not adding nodes.
