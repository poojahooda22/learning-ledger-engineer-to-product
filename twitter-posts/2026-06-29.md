Microsoft rewrote the TypeScript compiler in Go. VS Code is 1.5M lines.

Type-check time: 77 seconds to 7.5 seconds. Same codebase, same rules.

Why 10x and not 1.5x? Three real mechanisms:

1. Native binary, no JIT warmup. The old compiler was TypeScript running on V8. Every CI job paid V8's full JIT cost before checking a single type. Go starts instantly.

2. Goroutine parallelism. TypeScript 5.x was bounded by V8's single-threaded event loop. The Go rewrite checks independent files in parallel across all CPU cores.

3. Go's GC is built for large long-running heaps. V8's GC is tuned for web pages. Type-checking 1.5M lines is the wrong shape for V8.

What breaks: custom compiler plugins using the TypeScript Compiler API. That was a TypeScript-land interface. It no longer exists in the Go binary. Extension authors have until GA (mid-July) to update.

The real lesson: when a performance-critical tool is written in its own runtime, the runtime IS the ceiling. JIT tax, GC pauses, and single-threaded throughput are not things you algorithm your way out of. Sometimes you have to go below the language.

Playwright (26K lines): 25s to 3s. Same 10x.
