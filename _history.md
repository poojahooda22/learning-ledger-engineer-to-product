# History

Running log of every teardown. Newest at the bottom. Never repeat a product+feature pair listed here.

| Date | Product | Feature | One-line insight |
|------|---------|---------|------------------|
| 2026-06-13 | Spotify | Discover Weekly | Keep the expensive thinking offline and static (matrix factorization + Annoy index), serve the live path as a cheap memory-mapped lookup; the Monday refresh manufactures a weekly habit loop. |
| 2026-06-14 | Uber | Surge pricing | Turn "where" into one integer (H3 hex index), count supply/demand per hexagon via streaming + geo-sharding, then precompute and cache so the rider's live request is a single keyed lookup; hexagons beat squares because all 6 neighbors are equidistant. |
| 2026-06-15 | WhatsApp | Delivery receipts (the ticks) | Phones never talk directly; the server is a post office that does receive, look up, forward, ack, delete. Each tick (sent, delivered, read) is a tiny idempotent status event keyed by message id, relayed back the same path; store-and-forward queues plus 2M+ connections per Erlang box keep the hot path five steps long at any scale. |
| 2026-06-16 | Google Search | Autocomplete (query suggestions / typeahead) | Split the work into matching then ranking: walk a prefix trie (cost = letters typed, not catalog size) to the candidates, then rank ~10 survivors. Precompute top-k per node offline from query logs so the live keystroke is a cached lookup, not a sort over millions; shard by prefix, replicate for reads, prune branches that cannot beat the current top-k (PruningRadixTrie: up to 1000x faster). |
| 2026-06-17 | Zepto | Dark-store inventory and order routing | Three hard problems stitched together: point-in-polygon to pick the serving store (spatial index, not a per-store scan), atomic check-and-decrement so the last carton of milk never oversells (Lua DECR on a single owner, reserve-then-commit with TTL), and in-memory NEARBY to assign the closest free rider (match then rank). Shard inventory by store for free horizontal scale; split one hot SKU counter into N sub-counters under sale load; expensive forecasting runs offline, the live path is a cheap atomic reservation. |
| 2026-06-18 | Figma | Multiplayer cursors and live presence | One feature, two opposite data problems. The document must be permanent and conflict free: a tree of objects (ID + property hash map), resolved by property-level last writer wins with one server per document as the order referee (no timestamps needed); child order via fractional indexing (base-95 strings), reparenting via a parent property to keep identity. The cursors must be fast and disposable: ephemeral presence broadcast over the same WebSocket but never journaled or written to S3, throttled to ~30 Hz on the wire and interpolated to 60 fps in requestAnimationFrame, capped at ~200 visible cursors because past that the cost is quadratic and the value is zero. |
| 2026-06-19 | Netflix | Personalized artwork (per-member thumbnails) | Two halves: candidate generation (AVA harvests good frames offline, annotates visual/contextual/composition metadata, chunked-parallel on Archer) and selection (a contextual bandit picks 1 of N images live). Reward is the "take fraction": a quality play, not a click. Live path is cheap: a few dot products and an argmax over a short array, constant in catalog size, member context precomputed, pick cached per session for stability. The killer constraint is Replay off-policy evaluation: only ~1/N of uniform-random logs match a new policy's pick, so the candidate set N is kept small on purpose; small action space is what makes learning measurable. |
