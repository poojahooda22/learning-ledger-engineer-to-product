# History

Running log of every teardown. Newest at the bottom. Never repeat a product+feature pair listed here.

| Date | Product | Feature | One-line insight |
|------|---------|---------|------------------|
| 2026-06-13 | Spotify | Discover Weekly | Keep the expensive thinking offline and static (matrix factorization + Annoy index), serve the live path as a cheap memory-mapped lookup; the Monday refresh manufactures a weekly habit loop. |
| 2026-06-14 | Uber | Surge pricing | Turn "where" into one integer (H3 hex index), count supply/demand per hexagon via streaming + geo-sharding, then precompute and cache so the rider's live request is a single keyed lookup; hexagons beat squares because all 6 neighbors are equidistant. |
| 2026-06-15 | WhatsApp | Delivery receipts (the ticks) | Phones never talk directly; the server is a post office that does receive, look up, forward, ack, delete. Each tick (sent, delivered, read) is a tiny idempotent status event keyed by message id, relayed back the same path; store-and-forward queues plus 2M+ connections per Erlang box keep the hot path five steps long at any scale. |
| 2026-06-16 | Google Search | Autocomplete (query suggestions / typeahead) | Split the work into matching then ranking: walk a prefix trie (cost = letters typed, not catalog size) to the candidates, then rank ~10 survivors. Precompute top-k per node offline from query logs so the live keystroke is a cached lookup, not a sort over millions; shard by prefix, replicate for reads, prune branches that cannot beat the current top-k (PruningRadixTrie: up to 1000x faster). |
| 2026-06-17 | Zepto | Dark-store inventory and order routing | Three hard problems stitched together: point-in-polygon to pick the serving store (spatial index, not a per-store scan), atomic check-and-decrement so the last carton of milk never oversells (Lua DECR on a single owner, reserve-then-commit with TTL), and in-memory NEARBY to assign the closest free rider (match then rank). Shard inventory by store for free horizontal scale; split one hot SKU counter into N sub-counters under sale load; expensive forecasting runs offline, the live path is a cheap atomic reservation. |
| 2026-06-18 | Figma | Multiplayer cursors and live presence | One feature, two opposite data problems. The document must be permanent and conflict free: a tree of objects (ID + property hash map), resolved by property-level last writer wins with one server per document as the order referee (no timestamps needed); child order via fractional indexing (base-95 strings), reparenting via a parent property to keep identity. The cursors must be fast and disposable: ephemeral presence broadcast over the same WebSocket but never journaled or written to S3, throttled to ~30 Hz on the wire and interpolated to 60 fps in requestAnimationFrame, capped at ~200 visible cursors because past that the cost is quadratic and the value is zero. |
| 2026-06-19 | Netflix | Personalized artwork (per-member thumbnails) | Two halves: candidate generation (AVA harvests good frames offline, annotates visual/contextual/composition metadata, chunked-parallel on Archer) and selection (a contextual bandit picks 1 of N images live). Reward is the "take fraction": a quality play, not a click. Live path is cheap: a few dot products and an argmax over a short array, constant in catalog size, member context precomputed, pick cached per session for stability. The killer constraint is Replay off-policy evaluation: only ~1/N of uniform-random logs match a new policy's pick, so the candidate set N is kept small on purpose; small action space is what makes learning measurable. |
| 2026-06-20 | Stripe | Idempotency keys | A failed network call is not a failed payment; the server cannot tell "charge succeeded, reply lost" from "charge never happened," so retry is a coin flip between double charge and lost sale. The fix is not a seen-keys lookup but a forward-only state machine (started -> charge_created -> finished) checkpointed in Postgres, because the card charge is a foreign state mutation you cannot ROLLBACK. Atomic phases bracket the foreign call; a unique index on (user_id, key) plus a locked_at column plus serializable transactions stop concurrent retries (409 Conflict); reaper bounds storage at 24h; shard by account for linear scale since each key is self-routing and needs no global lock. Free artifact cache falls out for identical requests. |
| 2026-06-21 | Gmail | Smart Compose (inline sentence completion) | No catalog to match, so matching becomes generation: a neural LM plus beam search over a heap, not a trie. The whole feature lives under a per-keystroke latency budget (~100ms target, ~60ms p90), so they shipped the FAST model (LM-A: bag-of-words context fed into an RNN-LM) over the more accurate seq2seq-with-attention, encode the unchanging context once and cache it, only recompute the prefix live. TPUs cut serving from hundreds of ms (CPU) to tens of ms; a per-user n-gram model blended by linear interpolation adds personalization without per-user neural retraining; a confidence threshold lets it stay silent, which protects the habit loop and doubles as a load shedder at 1.5B users. |
| 2026-06-22 | YouTube | Recommendations (home feed and "watch next") | Recommendation is search with the same two halves. Candidate generation is matching: pose "what next" as extreme multiclass softmax over a million video classes, train with negative sampling, learn a user vector and a video vector per item, then at serving throw away the softmax and just do approximate nearest-neighbor search in dot-product space (billion videos down to a few hundred, sub-linear, no scan). Ranking is precision on the shortlist: a rich model that optimizes expected watch time, not clicks (weighted logistic regression in 2016; MMoE multitask + a shallow tower that learns and removes position/selection bias in 2019). The "example age" feature kills the old-video bias so fresh uploads surface. All whole-corpus work is offline (1B params, 100Bs of examples, embeddings + ANN index); the live path is one user vector, one nearest-neighbor lookup, one small sort. Autoplay + watch-time objective build the loop that drives ~70% of watch time. |
| 2026-06-23 | Amazon | Product search ranking (the results page) | One ambiguous word against 350M+ items is two problems, not one: matching (cheap candidate fetch from an inverted index, posting lists merged by typed words so cost tracks query length not catalog size, plus a two-tower semantic model whose precomputed product vectors fix the sneakers-equals-running-shoes gap via nearest-neighbor) then ranking (gradient boosted trees, pairwise NDCG objective, leaning on behavioral signals because near-identical product text cannot separate listings, with day-to-day position-bias correction so the top result does not freeze and new products still climb). Per-category rankers blended in All Product Search. The candidate-set size (a few thousand) is the dial that decouples ranking cost from catalog size; shard the index + scatter-gather + cache hot queries + push all training/indexing offline to survive ~60k searches/sec. |
| 2026-06-25 | Notion | The "/" slash command (and the block model it writes into) | The slash menu is a thin UI over one big idea: everything is a block (UUID + type + properties hash map + ordered content array of child IDs + parent), so a 60-item insert menu needs zero per-type storage paths and "insert" is just minting a UUID and splicing one ID into the parent's array. The same uniformity is what made it shard: every block belongs to exactly one workspace, so partition by workspace_id (480 logical shards, 32 boxes in 2021 -> 96 in 2023, same 480 shards so no re-partition, peak CPU 90% -> 20%), keep the single-workspace path on the shards and push whole-corpus scans to a CDC data lake (Debezium -> Kafka -> Spark -> S3/Hudi). Writes are operations grouped into transactions, applied optimistically to a local RecordCache before the server hears about them; audit-log migration beat logical replication for the cutover. |
| 2026-06-24 | Swiggy | Live order tracking (the moving rider and the ETA timer) | A rider's live location is an event stream, not a database row: pings flow phone -> Kafka -> processor -> Redis (latest position only) -> WebSocket push, so the live path never writes to the primary DB and a customer query is a keyed memory lookup. Raw GPS lies (20-40m jitter), so the server map-matches the trail to the OSM road graph with GraphHopper (HMM + Viterbi), and the client interpolates 60fps smoothness on top of a position that truly updates only every 3-5s (sparse truth, smooth fiction). The "Arriving in X min" is not one model but four refreshed legs (O2A, FM, WT, LM), each fed by restaurant stress, item mix, rider availability, and live speed measured from the fleet's own GPS pings; the rider fleet is the traffic-probe network. Decompose the one number so each leg can be measured and fixed. Geohash buckets in memory for nearest-rider; partition by city, precompute routing (contraction hierarchies) offline so the live query stays sub-millisecond. |
